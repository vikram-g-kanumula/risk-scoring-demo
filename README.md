# US Homeowners Risk Scoring Engine
### BD Capability Demo

**A GLM-first risk scoring architecture that captures compounding peril risk through an ML interaction layer trained on GLM residuals — with exact regulatory transparency at every layer.**

---

## Quick Start

```bash
# 1. Unzip and enter the project directory
unzip risk-scoring-demo.zip
cd risk-scoring-demo

# 2. Create a virtual environment (recommended)
python -m venv venv
source venv/bin/activate        # macOS / Linux
# venv\Scripts\activate         # Windows

# 3. Install dependencies  (~2 min)
pip install -r requirements.txt

# 4. Generate data and train models  (~6–11 min, one-time only)
python setup.py

# 5. Launch the demo
streamlit run app.py
```

The app opens at `http://localhost:8501`.

## Architecture

The core claim of this demo — that compound-peril interactions are systematically under-priced by additive models — is supported by a specific, auditable pipeline. The GLM is the causal anchor; M̂ is defined entirely in terms of the GLM's residuals.

### The Four-Step GLM Improver Pattern

This architecture implements the Dai (CAS E-Forum, Spring 2018) GLM Improver pattern:

```
Step 1 — GLM Baseline (Regulatory Layer)
  PoissonRegressor(log-link)  → λ_GLM  (annual claim frequency)
  GammaRegressor(log-link)    → μ_GLM  (expected severity | claim occurred)
  Baseline E[L] = λ_GLM × μ_GLM
  Features: 19 T1+T2 variables
  Output: DOI-auditable exp(β) rate relativities

Step 2 — GLM Residual (M̂ Training Target)
  M_actual(i) = claim_amount(i) / max(GLM_prediction(i), floor)
  M_actual > 1.0  →  GLM under-predicted; actual loss was higher
  M_actual = 1.0  →  GLM was exactly right
  M_actual < 1.0  →  GLM over-predicted (protective interactions)

Step 3 — M̂ Ensemble (Interaction Discovery Layer)
  Input: Tier 3 features only
         (wildfire_zone, roof_material, flood_zone, dist_to_coast_mi,
          construction_type, roof_age_yr, earthquake_zone, state,
          slope_steepness_pct, post_burn_rainfall_intensity)
  Models: RandomForest + HistGradientBoosting + ExtraTrees
  Meta:   Ridge + IsotonicRegression (OOF stacked)
  Target: M_actual — what the GLM left unexplained

Step 4 — Final Pricing
  E[L](i) = λ_GLM(i)  ×  μ_GLM(i)  ×  M̂(i)
  When M̂ = 1.0  →  reduces exactly to the pure GLM baseline
  Premium = E[L] / target_loss_ratio × expense_factor
```

### Display Artifact

A `TweedieRegressor(p=1.65)` is also trained and stored in `artifacts.pkl`. It is not used in the live pricing chain. It exists solely as the DOI rate-filing display artifact — its exp(β) coefficients are shown in the Methodology tab's regulatory comparison table. When an actuarial audience asks to see the filed model, this is what they see.

## What Changed from v2 to v3

The v2 codebase correctly framed the demo as "GLM-first with M̂ on top" but had three structural gaps that would be exposed by a carrier data scientist:

| Gap | v2 Behaviour | v3 Fix |
|-----|-------------|--------|
| **M̂ training target** | Trained on `M_true` (oracle multipliers hand-coded in data_generator) | Trained on `M_actual = claim_amount / GLM_prediction` — the real GLM residual |
| **Baseline anchor** | `predict_baseline()` called two HistGBM models (`freq_model × sev_model`) | `predict_baseline()` calls the actual Poisson × Gamma GLM |
| **Pricing formula** | `E[L] = λ_ML × μ_ML × M̂` (three independent ML models) | `E[L] = GLM_prediction × M̂` (GLM is the causal anchor, M̂ multiplies it) |

The architecture framing in `app.py` (narrative text, tab labels, SHAP section, footnotes) has been updated throughout to match what the code now actually does.

## The M̂ Table — What the Ensemble Learns

These are the compound-peril interactions that the M̂ ensemble is trained to predict from GLM residuals. The magnitudes reflect what the data shows the GLM systematically under-prices:

| Interaction | Representative M̂ | Actuarial Logic |
|-------------|------------------|----------------|
| Wood Shake × High Wildfire Zone | ×3.0–3.5 | Embers ignite degraded shingles at 572°F vs 700°F for asphalt. Camp Fire validation: 85% of structures that burned had WUI + aged roofs |
| High Flood Zone × Coastal <5mi | ×2.0–2.5 | Storm surge amplification. FEMA coastal surge studies: non-linear interaction between flood depth and coastal proximity |
| Wood Shake × Moderate Wildfire | ×1.8–2.1 | Partial ember transport distance; still significantly elevated vs baseline |
| Non-Wood × High Wildfire | ×1.5–1.8 | Even resilient roofs exposed in extreme fire behavior at WUI |
| High Earthquake Zone | ×1.3–1.6 | Seismic + structural age compound. Pre-1970 construction most sensitive |
| Old Roof (>20yr) × Frame Construction | ×1.2–1.4 | UV granule loss + combustible frame = water intrusion + fire cycle |
| Slope >55% × High WF × Post-Burn Rain | ×1.3–1.6 | Montecito 2018 archetype: Thomas Fire debris flow, 23 deaths, 400+ homes |
| Defensible Space ≥80 + Metal Roof | ×0.80–0.90 | Protective synergy — learned from data, not hard-coded |
| New Superior Construction (≤5yr) | ×0.82–0.92 | Modern code compliance + materials quality |

*Specific values depend on training run. The ranges above reflect expected M̂ from a residual-trained ensemble on this synthetic dataset.*

## Troubleshooting

**`KeyError: 'poisson_glm'` on app startup**
You are running app.py v3 against artifacts.pkl built by model_trainer.py v2. Re-run `python setup.py` to regenerate the pkl with v3 keys.

**`FileNotFoundError: models/artifacts.pkl`**
`setup.py` has not been run, or failed partway through. Check terminal output for the error and re-run after fixing.

**App loads but scores are unexpected / much lower than v2**
This is correct and expected behaviour. The v3 GLM baseline (Poisson × Gamma) produces different absolute E[L] values than the v2 HistGBM frequency/severity models. The relative ordering and tier assignments are correct. Score ranges for the 10 demo properties will differ from the v2 README values.

**`ModuleNotFoundError` on any import**
Run `pip install -r requirements.txt` before `python setup.py`.

**LightGBM install fails on Apple Silicon**
```bash
pip install lightgbm --no-binary lightgbm
# or
conda install lightgbm
```

**SHAP import error / SHAP values show as unavailable in Tab 5**
Ensure `shap>=0.44.0`. The GLM LinearExplainer and the M̂ TreeExplainer are loaded lazily inside Tab 5 — if shap is missing, Tab 5 falls back to model-native feature importances without crashing the rest of the app.

**`setup.py` finishes but `m_hat_vs_oracle_corr` metric is below 0.70**
This means the M̂ ensemble is not recovering the interaction structure well from GLM residuals. Most common cause: the GLM itself is under-fitting (check `glm_r2` in the metrics printout). If `glm_r2 < 0.40`, the residuals will be too noisy for the ensemble to learn from. Verify `data/homeowners_data.csv` was generated by `data_generator.py` v6 and not an older version.
